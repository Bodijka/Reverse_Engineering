Учебный проект по реверс-инжинирингу техники **WinAPI Hashing**.  
В работе исследуется бинарный файл `PE3.exe`, который скрывает используемые WinAPI-функции и получает их адреса во время выполнения через собственные аналоги `GetModuleHandle` и `GetProcAddress`.

## О проекте

Обычная таблица импорта у бинарника практически не раскрывает полезной информации. Вместо прямых импортов программа использует числовые хэши:

- хэш имени DLL;
- хэш имени API-функции;
- ручной поиск модуля через структуры процесса;
- ручной разбор export table у найденной DLL.

Такой подход усложняет статический анализ, потому что в бинарнике нет обычных строк вроде `CreateFileW`, `WriteConsoleA` или `SetFileInformationByHandle`.

## Инструменты

В ходе анализа использовались:

- **IDA Pro** для просмотра псевдокода и структуры программы;
- **x64dbg** для динамического анализа и проверки возвращаемых адресов;
- просмотр секции `.rdata` для поиска сохраненных хэшей.

## Основная идея WinAPI Hashing

В программе были найдены две важные внутренние функции:

| Адрес | Назначение |
|---|---|
| `sub_1400010C0` | поиск DLL по хэшу имени |
| `sub_140001000` | поиск API-функции по хэшу имени |

По смыслу они заменяют стандартные WinAPI:

| Обычный WinAPI | Внутренний аналог |
|---|---|
| `GetModuleHandle` | поиск модуля по хэшу |
| `GetProcAddress` | поиск функции по хэшу |

## Почему отдельно рассматривалась `.rdata`

Большинство функций были подтверждены динамически в x64dbg: после работы внутреннего резолвера в `RAX` появлялся адрес API, и отладчик подписывал его именем функции.

Отдельно был найден хэш:

```text
0xA427F6B3
```

Он находился в секции `.rdata` рядом с уже опознанными хэшами API. В памяти значение хранится в little-endian виде:

```text
B3 F6 27 A4
```

После проверки тем же алгоритмом хэширования это значение было определено как:

```text
HeapAlloc
```

## Поведение бинарника

По результатам анализа программа:

1. находит нужные DLL по хэшам;
2. разрешает адреса WinAPI-функций по хэшам;
3. получает путь к собственному исполняемому файлу;
4. открывает собственный файл;
5. пытается выполнить операции, связанные с переименованием и удалением файла;
6. выводит сообщения в консоль;
7. делает задержку через `Sleep`.

## Назначение проекта

Репозиторий создан как отчет по лабораторной работе и демонстрирует:

- базовый анализ PE-файла;
- работу с x64dbg;
- поиск API по хэшам;
- восстановление скрытых импортов;
- подтверждение найденных функций через динамический анализ.

## Предупреждение

`PE3.exe` используется только в учебных целях.  
Запускать подобные бинарные файлы стоит в изолированной среде или на копии файла, так как программа содержит логику, связанную с удалением собственного исполняемого файла.
